# 01 — Extract @opencdd/models Package

## Goal

Create `opencdd/cdd-models-ts` as a standalone public npm package
(`@opencdd/models`) containing the TS value-object layer and validator
port currently at `cdd-data/editor/src/models/`.

## Background

Today the TS model layer lives at `editor/src/models/` and is consumed
by:

- `editor/` (directly, as source).
- `browser/` (via Vite path alias `@opencdd/models` →
  `../editor/src/models/index.ts`).

Both web apps are leaving cdd-data. The shared model layer must become
a versioned, published artifact so each app can pin and upgrade
deliberately rather than sibling-directory-coupling.

## Scope

**In the package:**

- Value objects: `Entity`, `Klass`, `Property`, `Unit`, `ValueList`,
  `ValueTerm`, `Relation`, `ViewControl`, `AliasTable`, `MetaClass`,
  `ClassType`, `PropertyDataTypeElement`, `ValueFormat`.
- Validator port: TS Runner + all Ruby rules R01–R14.
- Codegen outputs: `PropertyIds.generated.ts`,
  `MetaClasses.generated.ts`.
- Tree walkers: `ClassTree`, `RelationTree`, `CompositionTree`,
  `EffectiveProperties`.
- Utility ports: `Guid`, `InstanceRule`, `StructuredValues` (7
  parser/serializer pairs).
- Exporters: `JsonExporter`, `YamlExporter`, `MermaidExporter`,
  `CsvWriter` (JS-runtime-friendly ports of the Ruby exporters).
- Helpers: `rubyInspect`, `byEntityCode`, `parseStringList`,
  `entityConstructors`, `referenceKinds`, `codeFromIrdi` /
  `asEntityIrdi` / `asPropertyIrdi`.

**Out of the package (stays with editor):**

- Zustand stores (`databaseStore`, `historyStore`, `selectionStore`,
  `validationStore`, `uiStore`, `fileStore`) — editor-specific UI
  state.
- Database mutation commands — editor concern.

**Future in the package (when shipped):**

- Parcel workbook reader/writer (Phase 3.5b, SheetJS) — both apps
  will want it.

## Tasks

1. Create `opencdd/cdd-models-ts` repo (public).
2. Copy `cdd-data/editor/src/models/` contents into `src/`.
3. Audit for `editor/`-specific imports (Zustand stores, UI state) and
   either remove or move them back to the editor.
4. Set up build:
   - `tsup` for dual ESM/CJS output, or `tsc --build` with project
     references.
   - `exports` map in `package.json` for clean subpath imports
     (`@opencdd/models`, `@opencdd/models/validators`, etc.).
   - TypeScript `.d.ts` emission alongside JS.
5. Set up Vitest config; port the existing model + validator specs.
6. Configure npm publish workflow:
   - GitHub Actions on tag push (`v*`) → `npm publish` (public).
   - Use `--provenance` for supply-chain attestation.
   - Build before publish; run tests as a gate.
7. Cut `v0.1.0` initial release.
8. Add `README.md` documenting public API surface + semver policy +
   codegen provenance (files are generated by cdd-data, see
   TODO.astro/02).

## Dependencies

- Blocks: 02 (submodule), 03 (cdd-data cleanup), 05+ (Astro browser).
- Blocked by: nothing.

## Acceptance criteria

- `npm install @opencdd/models` works from a clean checkout.
- Both ESM and CJS consumers can import all public symbols.
- `npm run build` produces typed `.d.ts` alongside JS.
- All existing model + validator specs pass under the new layout.
- `v0.1.0` published with provenance.
- No editor-specific dependencies (Zustand, etc.) leak into the
  package.
